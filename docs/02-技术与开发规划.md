# 非遗建筑主题拼图小游戏技术与开发规划

## 1. 当前工程现状

当前项目已经是一个可在 IDEA 中直接打开的 Maven 工程，基础环境如下：

1. `Spring Boot 4.0.6`
2. `Java 21`
3. `Maven Wrapper`
4. 当前只有启动类，还没有业务代码

这意味着我们可以直接在现有工程上继续扩展，无需重新建项目。

## 2. 技术路线建议

## 2.1 第一版推荐方案

建议采用“单工程前后端一体”的实现方式：

1. 后端：`Spring Boot`
2. 页面模板：`Thymeleaf`
3. 前端交互：`HTML + CSS + 原生 JavaScript`
4. 本地进度存储：`localStorage`
5. 图片与关卡数据：项目静态资源或本地 JSON 配置

推荐这个方案的原因：

1. 适合在 IDEA 中直接开发和运行。
2. 不需要额外引入 Node、打包工具和前后端分离流程。
3. 对于拼图小游戏这类中小型项目，开发成本更低。
4. 后续如果想升级为前后端分离，也容易平滑演进。

## 2.2 暂不推荐的第一版方案

第一版不建议一开始就使用：

1. React 或 Vue 全家桶。
2. 数据库。
3. 用户系统。
4. 复杂后台管理。

这些内容会显著增加工程复杂度，但对第一版核心目标帮助有限。

## 3. 推荐目录结构

后续开发建议整理为以下结构：

```text
feiyi-game
├─ src/main/java/com/example/feiyigame
│  ├─ controller
│  ├─ service
│  ├─ model
│  ├─ dto
│  └─ config
├─ src/main/resources
│  ├─ templates
│  │  ├─ index.html
│  │  ├─ levels.html
│  │  └─ game.html
│  ├─ static
│  │  ├─ css
│  │  ├─ js
│  │  ├─ images
│  │  └─ audio
│  └─ data
│     └─ levels.json
└─ docs
```

目录职责建议：

1. `templates`：放页面模板。
2. `static/css`：放页面样式。
3. `static/js`：放拼图逻辑、计时逻辑、页面交互逻辑。
4. `static/images`：放建筑原图、封面图、图标等。
5. `data/levels.json`：放关卡数据和知识说明。

## 4. 模块设计

## 4.1 后端模块

后端第一版职责尽量轻量，主要负责：

1. 返回页面。
2. 读取关卡配置。
3. 向前端提供关卡数据接口。

建议的后端模块：

1. `PageController`
   负责首页、选关页、游戏页跳转。

2. `LevelController`
   提供关卡列表、关卡详情等接口。

3. `LevelService`
   负责读取并整理关卡数据。

4. `Level`
   关卡实体，描述单关所需字段。

## 4.2 前端模块

前端第一版建议拆成以下逻辑模块：

1. `level-data.js`
   处理关卡信息读取与缓存。

2. `puzzle-engine.js`
   处理切片、打乱、交换、完成判定。

3. `timer.js`
   处理倒计时、超时结束、暂停恢复。

4. `progress.js`
   处理本地通关记录、最佳成绩、解锁状态。

5. `modal.js`
   处理通关说明弹窗和失败弹窗。

## 5. 数据结构建议

每个关卡建议具备如下字段：

```json
{
  "id": 1,
  "slug": "traditional-stage",
  "title": "传统戏楼",
  "coverImage": "/images/levels/stage-cover.jpg",
  "sourceImage": "/images/levels/stage-full.jpg",
  "difficulty": "easy",
  "rows": 3,
  "cols": 3,
  "timeLimitSeconds": 150,
  "description": "这里填写通关后展示的建筑简介。",
  "features": "这里填写建筑特色。",
  "culture": "这里填写历史文化价值说明。"
}
```

本地进度数据建议保存在浏览器中，结构可以类似：

```json
{
  "clearedLevels": [1, 2],
  "bestTimes": {
    "1": 98,
    "2": 143
  }
}
```

## 6. 页面与交互实现思路

## 6.1 首页

1. 以传统文化视觉风格做主视觉。
2. 一个“开始游戏”按钮进入选关页。
3. 一个“玩法说明”区域简述规则。

## 6.2 选关页

1. 用卡片列表展示关卡。
2. 卡片上显示封面、名称、难度、时限。
3. 已通关关卡显示标记。
4. 点击卡片进入游戏页。

## 6.3 游戏页

1. 上方显示关卡名、时间、操作按钮。
2. 中间显示拼图区。
3. 右侧或下方显示原图预览、玩法提示。
4. 通关时弹出介绍弹窗。

## 6.4 拼图交互建议

第一版推荐使用“拖拽交换”为主、“点击交换”为备选：

1. 桌面端优先支持拖拽交换。
2. 移动端如果拖拽体验不稳定，可退化为点击两块进行交换。

这样更利于兼容性和后续调试。

## 7. 开发阶段拆分

## 第一阶段：基础准备

目标：让工程具备页面开发基础。

任务：

1. 补充网页依赖。
2. 建立 `templates`、`static`、`data` 目录。
3. 确定项目名称、主题色、视觉方向。
4. 整理首批建筑图片和说明文案。

## 第二阶段：页面骨架

目标：完成首页、选关页、游戏页静态结构。

任务：

1. 首页布局。
2. 选关页卡片布局。
3. 游戏页拼图区布局。
4. 通关弹窗和失败弹窗样式。

## 第三阶段：核心玩法

目标：完成真正可玩的拼图逻辑。

任务：

1. 原图切片。
2. 随机打乱。
3. 拖拽或点击交换。
4. 完成判定。
5. 倒计时控制。

## 第四阶段：关卡与科普内容

目标：让游戏和知识内容连接起来。

任务：

1. 从配置文件加载关卡数据。
2. 按关卡渲染时间限制和难度。
3. 通关后显示对应介绍内容。
4. 支持下一关跳转。

## 第五阶段：体验优化

目标：提升完成度和展示效果。

任务：

1. 本地存档。
2. 最佳成绩。
3. 音效和动画。
4. 响应式适配。
5. 图片压缩与性能优化。

## 8. IDEA 开发建议

在 IDEA 中开发时建议这样推进：

1. 以 Maven 项目方式打开当前工程。
2. 确认项目 SDK 使用 `Java 21`。
3. 使用 `mvnw` 或 IDEA Maven 工具窗口管理依赖。
4. 使用 Spring Boot Run Configuration 启动项目。
5. 页面模板和静态资源修改后，及时通过浏览器验证样式和交互。

如果后续我们开始编码，优先顺序建议是：

1. 先搭页面骨架。
2. 再写拼图引擎。
3. 最后接通关卡数据和通关弹窗。

## 9. 风险与注意事项

## 9.1 内容风险

1. 建筑是否适合归类到“非遗”需要提前统一口径。
2. 建筑介绍文案要注意准确性，避免概念混用。

## 9.2 资源风险

1. 图片版权必须明确。
2. 图片尺寸过大会影响加载速度。

## 9.3 交互风险

1. 网页拖拽在不同设备上的体验差异较大。
2. 时间限制需要多次试玩才能调到合理区间。

## 10. 第一版验收目标

建议把第一版定义为“可演示版本”，达到以下结果即可：

1. 本地启动后能进入首页、选关页和游戏页。
2. 至少 4 个关卡可以正常游玩。
3. 每关有独立时间限制。
4. 通关后有对应建筑介绍弹窗。
5. 关卡完成状态可保存在本地。

## 11. 下一步建议

在开始编码之前，最值得先定下来的有三件事：

1. 项目最终名称和主题口径。
2. 第一批使用哪些建筑图片。
3. 第一版是否采用 `Thymeleaf + 原生 JavaScript` 的轻量方案。

如果这三项确认下来，后面就可以很顺地进入页面骨架和功能开发阶段。
